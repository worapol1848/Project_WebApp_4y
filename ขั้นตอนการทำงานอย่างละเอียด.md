# 🔄 ขั้นตอนการทำงานและ System Architecture (เจาะลึกการแตกฝั่ง Frontend User/Admin)

เอกสารฉบับนี้อธิบายลอจิกการทำงานเชิงลึก (Deep Technical Flow) เพื่อใช้ในรายงานโปรเจกต์ระดับอุดมศึกษา โดยเจาะลึก **การทำงานที่แยกจากกันอย่างเด็ดขาดระหว่างผู้ใช้งานทั่วไป (User) และผู้ดูแลระบบ (Admin) บนระบบพอร์ทัล Frontend** พร้อมการเชื่อมโยงกับ Backend API และโครงสร้างฐานข้อมูลแบบละเอียดยิบ

---

## 🧑‍💻 1. การทำงานฝั่งผู้ใช้งานทั่วไป (User / Customer Portal)
*(โดเมนการใช้งาน: สมัครสมาชิก, เลือกซื้อ, ติดตามสถานะ, รายการโปรด และคอมเมนต์รีวิว)*

**ข้อจำกัดทางสิทธิ์การเข้าถึง (Access Control Limitations):** 
- ❌ ลูกค้า **ไม่สามารถ** สแกนใบหน้าได้ (ลายนิ้วมือ/ใบหน้าสงวนไว้สำหรับความมั่นคงระดับ Super Admin เท่านั้น)
- ❌ ลูกค้า **ไม่สามารถ** พิมพ์ไฟล์ใบเสร็จ (PDF) ชนิดเต็มรูปแบบได้ (สงวนสิทธิ์ให้แอดมินปริ้นท์สำหรับแนบลงกล่องพัสดุ ป้องกันการปลอมแปลง)
- ❌ ลูกค้า **ไม่มีสิทธิ์** เข้าถึงหน้าแดชบอร์ดหลังบ้านใดๆ

### 1.1 หน้าสมัครสมาชิกและเข้าสู่ระบบ (Auth Page)
- **Frontend Logic:**
  - ผู้ใช้งานสามารถสลับแถบ Login และ Register
  - ระบบบันทึก Token ในรูปของ JSON Web Token (JWT) ลงใน LocalStorage หลังล็อกอินสำเร็จ
- **API Flow:** `POST /api/auth/register` และ `POST /api/auth/login` (เข้ารหัสผ่านด้วย Bcrypt)

### 1.2 หน้าสินค้า (Home / Products / Product Detail) และ สิ่งที่อยากได้ (Wishlist)
- **Frontend Logic:**
  - แสดงแคตตาล็อกสินค้าพร้อม **Brand Filter** (ตัวกรองแบรนด์)
  - ผู้ใช้กดหัวใจเพื่อบันทึกสินค้าลงหน้า **Wishlist (`/wishlist`)** เพื่อเก็บไว้ดูทีหลัง
  - ในหน้ารายละเอียด (`/product/:id`) ผู้ใช้ต้องเลือก Size และ จำนวน ก่อนเพิ่มลงตะกร้า
  - **การรีวิว (Comments):** ลูกค้าที่ซื้อสินค้าไปแล้ว สามารถทิ้งรีวิวคอมเมนต์ไว้ที่หน้านี้ให้คนอื่นดูได้
- **API Flow:** `GET /api/products`, `POST /api/wishlist`, `POST /api/cart`
- **Database Tables:** `products`, `wishlist`, `product_comments`

### 1.3 หมวดการชำระเงินและปักหมุดที่อยู่ (Cart & Payment ` `/payment` `)
- **Frontend Logic:**
  - **แผนที่ (Map Locator):** ลูกค้าใช้แผนที่ (React-Leaflet) เลื่อนและปักหมุดที่อยู่ ระบบจะจับพิกัด (Latitude/Longitude) ทันที
  - **การขนส่ง:** ฟังก์ชันประเมินราคาตามประเภท (Messenger กทม. 60บ. / EMS ตจว. 150บ.)
  - **สลิป:** ต้องแนบไฟลล์ภาพ (Upload Slip) เพี่อยืนยัน
- **API Flow:** `POST /api/orders` (แนบฟอร์มข้ามโครงข่ายแบบ `multipart/form-data` ด้วยเครื่องมือ Multer)
- **Backend Transaction:** ดึงข้อมูลที่อยู่ลูกค้า ณ วินาทีนั้น มาเก็บบันทึกลงตารางออเดอร์ทันที (เพื่อทำ Address Snapshot) จากนั้นหักลบของออกจากสต๊อก และเคลียร์ตะกร้าทิ้ง 

### 1.4 หน้าติดตามสถานะและชดเชย (My Orders ` `/myorders` `)
- **Frontend Logic:**
  - ติดตาม Timeline (Pending > Shipped > Arrived)
  - **การรับสินค้า:** เมื่อสถานะถึง Arrived ลูกค้าต้องกด **"ฉันได้รับสินค้าแล้ว"** ยิงไปปิดสถานะเป็น Delivered
  - **ฟอร์มคืนเงิน (Refund Data):** กรณีแอดมินยกเลิกออเดอร์ (สินค้าหมด) ลูกค้าจะเห็นหน้าต่างให้กรอก "เลขบัญชีรับเงินโอนคืน" ตอบกลับไปให้แอดมิน
- **API Flow:** `PUT /api/orders/:id/deliver` และ `PUT /api/orders/:id/refund-info`

## 💼 2. การทำงานฝั่งผู้ดูแลระบบ (Admin Portal)
*(โดเมนการใช้งาน: อนุมัติสลิป, จัดส่ง, พิมพ์เอกสาร, และอัปเดตสต๊อก)*

**ข้อกำหนด:** เข้าสู่ระบบด้วย Username/Password ปกติ และมองเห็นข้อมูลเชิงคลังสินค้าเท่านั้น

### 2.1 หน้าการจัดการคำสั่งซื้อ (Order Management ` `/admin/orders` `)
- **Frontend Logic & Workflow:**
  - **อนุมัติสลิป:** แอดมินดูรูปสลิปเทียบยอดเงินจริง แล้วกดยืนยัน (ยิง `PUT /api/orders/:id/verify-slip`)
  - **เปลี่ยนสถานะจัดส่ง:** แอดมินกดเปลี่ยนสถานะเป็น ส่งของแล้ว (Shipped) หรือ ของถึงปลายทาง (Arrived)
  - ✅ **การพิมพ์ใบอนุญาต (PDF Receipt):** แอดมินเป็นคนเดียวที่จะคลิกดึงไฟล์ออกมาทางแอปพลิเคชันรูปแบบ PDF (ผ่านไลบรารี `jsPDF`) สำหรับจัดการเอกสารออกสู่โลกจริง
  - **ระบบตีกลับ (Cancel & Refund):** หากสต๊อกบั๊ก แอดมินจะกดยืนยันการ Cancel ทิ้ง พร้อมระบุเหตุผล (ระบบหลังบ้านจะบวกสต๊อกคืนร้านทันที) และเมื่อโอนเงินคืนตามเลขบัญชีที่ขอมา ก็มากดปิดงาน "Refunded"

### 2.2 หน้าการแก้ไขข้อมูลสินค้าชั้นลึก (Products & Inventory)
- **Frontend Logic:**
  - การจัดการรูปภาพสินค้า ชื่อ และแบรนด์ (สร้างแบรนด์ใหม่ให้ตรงกับตัวกรอง)
  - แผงควบคุมจำนวนไซส์สินค้า (`/admin/inventory`) ที่อัปเดตแบบเรียลไทม์

### 2.3 การแสดงผลกราฟสถิติ (Revenue / Best Sellers)
- **Frontend Component:** นำ API จาก Backend ส่งเข้ากราฟไลบรารี **Recharts** สร้างการมองเห็นแนวโน้มเส้นกราฟ (Trend line) และวงกลมไข่เจียว (Pie Chart) เสนอเป็น Dashboard Management

## 👑 3. การทำงานฝั่งผู้จัดการสูงสุด (Super Admin Management)
*(โดเมนการใช้งาน: บริหารงานบุคคล, ตรวจจับทุจริตแบบ Deep Tech)*

### 3.1 การสแกนใบหน้าเข้าทำงาน (Biometric AI Login)
- **Frontend Flow:** เมื่อเลือก Role=SuperAdmin ระบบ React จะเรียกใช้งานกล้องเว็บแคมผ่านแพ็คเกจ `@vladmandic/face-api` ทันที
- **Machine Learning Layer:** สร้างกรอบหน้า และจับจุดพิกัดออกมาเป็นอาร์เรย์ (Vector Float Array) ยิงไปเทียบระยะ Euclidean Distance ในคลังข้อมูลหลังบ้าน หากเป็นเจ้าตัวจริงจึงจะจ่าย Authentication ตั๋วผ่านทาง (JWT Token) ให้ไปหน้าจัดการแอดมินได้

### 3.2 กรมการควบคุมและการตรวจสอบพฤติกรรม (Super Admin Dashboards)
- **หน้าจัดการแอดมิน (`/superadmin/manage`):** อนุมัติ จ้าง หรือลบแอดมินธรรมดาทิ้ง รวมถึงบังคับสแกนหน้าเข้าระบบใหม่
- **กล้องวงจรปิดระดับ Data (System Logs ` `/admin/logs` `):** มี Middleware ดักจับและเขียนบันทึกลงตาราง `system_logs` ตลอดเวลา แอดมินคนไหนแก้ไขสต๊อก หรือกดยกเลิกสินค้า ระบบจะทิ้งร่องรอยไว้หมด ซึ่งหน้าต่างนี้ มีเฉพาะซูเปอร์แอดมินเท่านั้นที่จะเปิดอ่านได้

## 🛠️ โครงสร้างความต้านทานระดับแอพพลิเคชัน (Technical Resilience Mechanism)

เพื่อรายงานโครงสร้างที่มีความปลอดภัยและเสถียร เรามีกระบวนการเพิ่มเติม:
1. **Frontend Isolation Route (React Guard):**
   มีการห่อหุ้ม `ProtectedRoute` ป้องกันการข้ามหน้าต่าง (Routing Barrier) ถ้าไม่มี JWT ฝั่ง Admin ต่อให้พิมพ์ URL หรือทาย URL ก็ไม่สามารถเห็นหน้า UI ได้ (React Router จะปัดกลับสู่หน้าแรก)
2. **Database Translation Layer (SQL Transaction):** 
   การสั่งซื้อที่มีการลดล้างสต๊อกจะใช้ `Transaction (beginTransaction)` คุมไว้ ถ้าไฟดับตอนดึงตารางใดตารางหนึ่ง ข้อมูลตะกร้าและการซื้อทุกอย่างจะ Rollback ทิ้งทั้งหมด ไม่มีการเหลื่อมล้ำข้อมูลสูญหาย (Data Integrity Constraint)
